## 〇、為什麼要這份規範

### Elsa 的工作型態

> 「我的購物 CRM 做起來差不多了,但我是補丁疊代超狂的人,難保不會出新招。」

UNI 是一人工作室,系統覆蓋 4 大系統(主系統 / 點數 / 尊爵 / 購物),Elsa 的工作型態是**高頻優化 + 不斷出新招**。

這種工作型態的最大風險不是「現在會崩」,而是:

> **動了某個東西,結果炸到別的地方。**

### 規範要解決的 3 個問題

1. **「我不知道這次改的東西會不會炸到別的」** → 4 層分層 + 3 問檢查
2. **「Claude 每次新對話要重新教」** → 規範文件作為永久 SOP
3. **「我不想停下來做標註」** → 漸進式標註 SOP

### 規範的鋼鐵原則

```
1. Claude 主判斷層級,Elsa 不用思考「這是哪一層」
2. 不重構現有系統,只貼標
3. 標註寄生在主線工作上,絕不停滯
4. 每次動工前,Claude 自動執行老鷹三檢查
```

## 一、四層架構總覽

```
┌─────────────────────────────────────────────────┐
│  🛡️ L0 回滾層 ROLLBACK(橫切所有層的保險機制)   │
│     - GAS 部署版本 / ENABLED 開關                │
│     - GitHub commit / 1 行接入碼                 │
│                                                 │
│  ┌──────────────────────────────────────────┐  │
│  │ 🔒 L1 資料層 DATA                         │  │
│  │    Sheet 欄位 / ID 規則 / API 介面契約    │  │
│  │    紀律:只增不減,不改名,不換順序        │  │
│  └──────────────────────────────────────────┘  │
│              ↑ 讀寫                             │
│  ┌──────────────────────────────────────────┐  │
│  │ ⚙️ L2 邏輯層 LOGIC                        │  │
│  │    業務計算 / 規則判斷 / 商業參數          │  │
│  │    紀律:改前過 3 問檢查                  │  │
│  └──────────────────────────────────────────┘  │
│              ↑ 觸發                             │
│  ┌──────────────────────────────────────────┐  │
│  │ 🧪 L3 行為層 BEHAVIOR                     │  │
│  │    LINE 互動 / 前端 / 行銷實驗            │  │
│  │    紀律:必須有 ENABLED 開關 + 獨立模組    │  │
│  └──────────────────────────────────────────┘  │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 層級依賴關係

```
L3 行為層  →  呼叫  →  L2 邏輯層  →  讀寫  →  L1 資料層
                                                  ↑
                                  L0 包覆所有層,提供回滾保護
```

### 關鍵原則:**反向不准、橫切不准**

- ✅ 上層可以呼叫下層(L3 → L2 → L1)
- ❌ 下層不准呼叫上層(L1 不可以反過來去呼叫 L2)
- ❌ 同層之間若有相依,要記錄到「跨模組依賴清單」

## 二、L1 資料層:鎖死,只增不減

### 定義

L1 = 系統的「命根」。動了 L1 等於整個系統根基動搖。

### 屬於 L1 的東西

| 類型 | UNI 具體例子 |
|---|---|
| Sheet 欄位名稱 + 順序 | 主系統 預約紀錄 A:預約編號 / B:姓名 / C:電話 / D:服務 / E:日期 / F:時間 |
| ID 編號規則 | `U21e198ba1e0185560050f87d7bedc7f0`(LINE UID 格式)/ `C0001`(顧客 ID) |
| 跨系統呼叫的 API action 名稱 | `getPoints` / `drawCharacter` / `replyLineMessages` |
| 全域變數命名 | `LINE_TOKEN`(主系統)/ `OWNER_UID` |
| Script Properties 的 key 名 | `CLAUDE_API_KEY` / `CLOUDINARY_CLOUD_NAME` |
| 跨系統共用的常數 | `POINTS_GAS_URL` / `MAIN_GAS_URL` |
| Sheet 工作表名稱(分頁名) | `預約紀錄` / `📇 顧客資料庫` / `📨互查診斷` |

### L1 鋼鐵紀律

```
✅ 可以做:
   - 新增欄位(在最後一欄加)
   - 新增 ID 類型(完全新的種類)
   - 新增 API action(用新名字)
   - 新增 Script Properties key

❌ 絕對不可以:
   - 刪除既有欄位
   - 改既有欄位名稱
   - 改既有欄位順序
   - 改既有 ID 規則(例如把 C 改 CUST)
   - 改既有 API action 名稱
   - 改既有全域變數名
   - 刪除既有 Script Properties key
```

### 為什麼這麼嚴?

因為 L1 一動,所有讀寫的地方全部要連帶改。今天的「LINE_CHANNEL_ACCESS_TOKEN vs LINE_TOKEN」事件就是經典案例 — 一個變數名沒對齊,LINE 訊息全部送不出去。

### 改 L1 的合法流程(必須按順序)

```
1. 先在規範文件「資料層欄位鎖死清單」加註「擬新增 XXX」
2. 列出所有會用到此資料的 L2 / L3 模組
3. 評估影響範圍(會不會牽動現有功能)
4. Claude 出「L1 變更影響評估報告」
5. Elsa 點頭
6. 才開始實作(L1 改完,L2/L3 配套改)
```

## 三、L2 邏輯層:可改,但要過 3 問

### 定義

L2 = 「資料怎麼算出來」的邏輯。改 L2 不會動到資料結構,但會改變系統行為。

### 屬於 L2 的東西

| 類型 | UNI 具體例子 |
|---|---|
| 業務計算公式 | 點數計算(消費 1 元 = 1 點)、抽獎機率(A 賞 5%) |
| 規則判斷 | 預約衝突檢查、跨系統互查、Coupon 折扣計算 |
| 商業參數 | 99 點抽獎門檻、譽望分級(實習/鋼鐵/傳奇)、儲值方案 |
| Flex 卡片內容 | 標題、按鈕文字、版面排列 |
| Rich Menu 切換邏輯 | hair / style / art 三套切換規則 |
| 提醒時機 | 預約前 24h 提醒、回購預測週期 |
| 文案內容 | LINE 自動回覆、警示訊息、引導文字 |

### L2 紀律

```
✅ 可以改:
   - 任何 L2 內容,只要不動到 L1 欄位/ID/API

⚠️ 改之前必須過「3 問檢查」(見第七章)

🛡️ 必須有回滾路徑(L0):
   - 改前先 commit / 部署版本紀錄
   - 重大改動先做 L3 試
```

### L2 vs L1 的灰色地帶

**判斷準則:**

> 「這次改動,會不會讓其他模組讀到舊欄位時報錯?」
>
> - 會 → L1
> - 不會 → L2

範例:
- 改「點數計算公式從 1:1 變 1.5:1」 → **L2**(不影響欄位結構)
- 改「點數欄位名從 balance 變 points」 → **L1**(所有讀的地方都要改)

## 四、L3 行為層:亂試區,壞了不影響核心

### 定義

L3 = 「實驗區」。Elsa 的「補丁疊代狂」全部丟這層,合法亂試。

### 屬於 L3 的東西

| 類型 | UNI 具體例子 |
|---|---|
| 新關鍵字攔截 | 點數 Bot 攔截(`m_points_bot.gs`)、評論 AI、申訴關鍵字 |
| 新 LINE 文案玩法 | 限時活動推播、節日問候、生日提醒 |
| 行銷實驗 | 沉睡客喚醒、回購預測、AB 測 |
| 新前端頁面 | `points-game.html`、未來的 `kid-game.html` |
| 新 Flex 模板 | 服務選單第 4 頁、儲值引導卡 |
| 短期活動 | 母親節、週年慶、聖誕節活動 |

### L3 紀律(對齊 Elsa 既有「斷路器三項」)

```
✅ 必須:
   1. 有 ENABLED 開關(可一鍵關閉)
   2. 是獨立模組(不跟其他 L3 黏在一起)
   3. 1 行接入碼(主系統只貼 1 行,刪掉就斷開)

✅ 鼓勵:
   - 大膽試新招
   - 失敗就關掉
   - AB 測比較版本

⚠️ 警告:
   - L3 不可以直接改 L1 資料層(必須透過 L2)
   - L3 出包必須立刻能關,不准影響 L1/L2
```

### L3 模組標準骨架

```javascript
/* ═════════════════════════════════════════
 * 🧪 L3 行為層:[模組名稱] vX.X
 * ═════════════════════════════════════════
 * 📁 部署位置:[哪個 GAS 專案]
 * 📌 接入位置:[哪個檔案的哪一行]
 * 🚨 緊急回滾:刪掉接入碼 + 重新部署
 * ⚠️ 此模組不可直接寫入 L1 資料層
 * ═════════════════════════════════════════ */

const MODULE_CONFIG = {
  ENABLED: true,  // 🛡️ L0 回滾開關
  // ...
};

function moduleEntry(event) {
  if (!MODULE_CONFIG.ENABLED) return false;
  // ...
  return true;
}
```

## 五、L0 回滾層:橫切所有層的保險機制

### 定義

L0 不是垂直堆疊的「層」,而是**橫切 L1/L2/L3 所有層的保險機制**。

每次動 L1/L2/L3,都自動有 L0 保護。

### Elsa 已具備的 L0 機制(只是命名 + 規範化)

| 機制 | 對應層級 | 已用於 |
|---|---|---|
| GAS 部署版本(v193, v194, v195) | 所有層 | 主系統、點數系統、尊爵 |
| `ENABLED: true/false` 開關 | L3 | m_points_bot、各種行為層 |
| 1 行接入碼(`if typeof... && xxx(event)) return;`) | L3 接入 L2 的點 | 主系統 handleMessage |
| GitHub commit history | 所有 HTML | booking.html、points-game.html |
| Sheet 備份 | L1 資料層 | 重大改動前手動備份 |
| Script Properties Backup | L1 環境變數 | 改 token 前 |

### L0 規範化(寫成 SOP)

```
動 L1 之前:
   ✅ 先在 Sheet 做副本(整份複製分頁)
   ✅ 在「資料層欄位鎖死清單」紀錄變更
   ✅ Claude 出「影響評估報告」

動 L2 之前:
   ✅ 過 3 問檢查
   ✅ 確認部署版本可回(看一下 GAS 部署列表)
   ✅ 高風險:先做 L3 試 → 通了再正式改 L2

動 L3 之前:
   ✅ 確認 ENABLED 開關存在
   ✅ 確認接入碼可一行刪除
   ✅ 確認新部署版本可回到上一版
```

## 六、標註語法規則

### 統一格式

```javascript
// [L1] 這行屬於資料層
// [L2] 這行屬於邏輯層
// [L3] 這行屬於行為層
```

**不使用:**
- ~~`// [DATA_LAYER]`~~(太長)
- ~~`// 🔒 L1`~~(emoji 在不同編輯器顯示不一致)

### 標註位置(三種粒度)

#### 粒度 A:檔案頂部(必須)

```javascript
/**
 * ════════════════════════════════════════
 * 🧪 [L3] m_points_bot.gs · LINE 點數攔截 v1.3.1
 * ════════════════════════════════════════
 * 屬於:行為層(L3)
 * 接入:主系統 程式碼.gs handleMessage 第 1306 行
 * 依賴:L2 邏輯層(_callPointsApi_)/ L1 資料層(POINTS_GAS_URL)
 * 回滾:POINTS_BOT_CONFIG.ENABLED = false
 * ════════════════════════════════════════
 */
```

#### 粒度 B:區塊頂部(常用)

```javascript
/* ═══ [L1] 資料層:Sheet 欄位定義 ═══ */
const COL = {
  NAME: 1,
  PHONE: 9,
};

/* ═══ [L2] 邏輯層:點數計算 ═══ */
function calcPoints(amount) { ... }

/* ═══ [L3] 行為層:LINE 關鍵字攔截 ═══ */
function routePointsBot(event) { ... }
```

#### 粒度 C:單行標註(極少用)

```javascript
const MAX_DRAW_PER_DAY = 5;  // [L2] 商業參數
```

### 標註的時機(對齊「漸進式」原則)

```
✅ 一定要標:
   - 新建檔案(從第一行就分層)
   - 動到既有檔案的某個區塊(順手把這個區塊標一下)
   
❌ 不需要標:
   - 沒動到的區塊
   - 純查看的檔案(不動就不標)
```

## 七、3 問檢查清單(改 L2 必過)

改 L2 之前,Claude 主動問你這 3 件事(你不用想,只要回答):

### 問 1:「會不會動到 L1?」

```
具體問法:「這次改的東西,會不會用到新欄位 / 新 ID 規則 / 新 API 名稱?」

→ 會  → 停手!先走 L1 變更流程(見第二章)
→ 不會 → 過第二問
```

### 問 2:「有沒有回滾路徑?」

```
具體問法:「現在的部署版本是 vXXX,改完是 vXXX+1,可不可以一鍵回?」

→ 不能 → 停手!先建立回滾點
→ 能  → 過第三問
```

### 問 3:「要不要先做 L3 試?」

```
具體問法:「這個改動的風險高嗎?如果直接改 L2,出問題會影響哪些顧客?」

→ 高風險(影響核心、影響金流、影響預約)
   → 先做 L3 試(獨立模組、ENABLED 開關)
   → 試通了再正式改 L2
   
→ 低風險(微調文案、改個顏色、加個欄位顯示)
   → 直接改 L2
```

### 3 問檢查的「快速版」

如果是 Claude 一眼判斷出來「絕對是低風險」(改個 emoji、改個按鈕顏色、改個提示文字),可以跳過 3 問,但要明確告訴 Elsa「這次跳過 3 問,理由:XXX」。

## 八、Claude 動工前自檢清單(老鷹三檢查)

任何動工(改 .gs / 改 .html / 改 Sheet / 部署)之前,Claude 主動執行:

### 🦅 檢查 1:這次要動的是什麼?

```
☐ 要動哪個檔案?
☐ 要動哪一段?
☐ 屬於哪一層(L1 / L2 / L3)?
☐ 預估影響範圍?
☐ 預估時間?
```

### 🦅 檢查 2:這次「不」做什麼(避免越線)?

```
☐ 不會順手改其他檔案
☐ 不會升級不相關的東西
☐ 不會主動標註沒動到的區塊
☐ 不會引入新依賴
```

### 🦅 檢查 3:還有什麼要跟 Elsa 對齊?

```
☐ 商業邏輯確認(有沒有商業鐵則被觸碰?)
☐ 公版抽離(這次改動會不會破壞 // [UNI_EXCLUSIVE] 標註?)
☐ 回滾方法已就緒?
☐ 文件位置正確?(.md → docs/,.gs → GAS 內,.html → GitHub Pages)
```

### 三檢查通過後才動工

如果任何一項不確定,Claude 必須先問 Elsa,絕不擅自動工。

## 九、漸進式標註 SOP(寄生主線)

### 鐵則:**不為標註而停下來**

### 工作模式

```
情境 A:Elsa 說「要做 X 新功能」
  → Claude 先做老鷹三檢查
  → 動工時,順手把【這次碰到的區塊】標一下
  → 沒碰到的區塊,永遠不主動標

情境 B:Elsa 說「要修 Y 函式」
  → Claude 打開該函式
  → 如果這個函式還沒貼過層級註解 → 順手標一下(5 分鐘)
  → 修完就走,不順便標旁邊的函式

情境 C:Elsa 說「我想看 Z 是怎麼運作的」
  → Claude 開檔案查看
  → 不主動標(因為沒要改)
  → 除非 Elsa 說「順便標一下」

情境 D:新模組(從零開始)
  → 100% 嚴格 4 層
  → 從第一行就有 // [L?] 註解
  → 檔案頂部有完整層級說明
```

### 漸進式 vs 專案化的差異

| 項目 | 專案化標註 | 漸進式標註(本規範採用) |
|---|---|---|
| **時間** | 集中 5-7 小時 | 分散到數月,每次 5-10 分 |
| **主線停滯** | 🔴 有 | 🟢 無 |
| **遺忘風險** | 🟡 中 | 🟢 無 |
| **覆蓋率** | 🟢 100% | 🟡 80%(只標常動的) |

## 十、公版抽離 SOP 升級版(整合 4 層)

### 原 SOP(三段式)

